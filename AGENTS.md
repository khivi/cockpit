# Privacy & Internal References

This is a public repository. Never include the following in commits, PRs, code comments, or documentation:

- Internal ticket IDs (Linear `ENG-123`, Jira `PROJ-456`, etc.)
- Internal GitHub PR/issue URLs from private repos
- Real names of teammates (use roles instead: "the reviewer", "the on-call engineer")
- Internal Slack channels, wiki URLs, or tool links
- Internal hostnames, service names, or infra identifiers
- Customer names or company-specific identifiers

When writing commit messages or PR descriptions:

- Describe *what* changed and *why*, not which ticket tracks it
- Reference public GitHub issues only (`#123` in this repo)
- If context requires an internal ticket, summarize the requirement instead of linking

Before committing, scan for cases gitleaks can't catch:

- Your team's real ticket prefixes (e.g. `ENG-123`, `LIN-456`)
- `@firstname` references that aren't GitHub handles

## Worktree discipline

Always use a dedicated git worktree for any code change. Never commit directly to `main`/`master` in the primary checkout, and never make in-place edits on a feature branch without a dedicated sibling worktree.

**Why**: The cmux + cockpit workflow keys off the one-worktree-per-branch invariant. In-place edits on the primary checkout pollute its `main` (which must always equal `origin/main`) and break PR-tracking — cockpit derives per-branch state from `git worktree list` and misattributes or drops cells for any branch not isolated in its own worktree.

**How to apply**: Before any Edit or Write, run `git branch --show-current` and `git worktree list`. If HEAD is `main`/`master`, or if the working-tree path is the primary checkout (first entry in `git worktree list`), stop and spawn a worktree via `/cockpit:new` before touching any file.

## Architecture notes

**`docs/state-machine.md` visualizes this section — keep it in sync.** Four Mermaid diagrams map the three state sources (GitHub PR, Claude session, cmux workspace — plus an auxiliary Linear read) onto the decision logic: orientation map, reconcile decision tree (slow tick), nudge idle-gate, and cell data-flow/ownership. Any code change that alters that logic — the decision functions (`match_worktrees`, `_spawn_missing_workspaces`, `nudge_if_idle`, `_track_dev_done`, `_maybe_autoclose`), the cell writers in `cache.py`, tick cadence/ownership, or the spawn/teardown/nudge/devdone/color rules — MUST update the matching diagram in the same PR. Treat a stale diagram like a stale comment: it is worse than none. The diagrams render on GitHub from the blob view.

**Worktree + workspace inventory is derived, not stored.** Each cycle re-reads `git worktree list` and `cmux tree` rather than maintaining its own `state.json`. PR payloads *are* cached (`~/.config/cockpit/cache/<repo>__pr-<N>.json`) because they're a network round-trip; everything else is recomputed. Don't add a `state.json` for worktree/workspace identity — drift between cached identity and the real `git`/`cmux` state was the bug class this design avoids.

**Packaged as `cockpit` (console entry point), not invoked by file path.** The import package is `cockpit/`; `[project.scripts] cockpit = cockpit.cli:main` is a thin subcommand dispatcher (`cockpit/cli.py`) that lazily routes `watch/setup/statusline/starship/new/nudge` to each module's `main` (`watch` and `setup` both route into `cockpit.cockpit` as `--watch`/`--setup`; `setup` re-wires the statusLine config without starting the daemon). The plugin's `commands/*.md` and `hooks/hooks.json` invoke `cockpit <sub>`; the generated statusline/starship configs use `{python} -m cockpit.cli <sub>` (interpreter + module dispatch, so they resolve without `cockpit` on PATH in starship's render env). The wheel version is single-sourced from `.claude-plugin/plugin.json` via a hatch version hook, so the pre-push bump still drives it. Because the plugin calls the `cockpit` console script, the package must be installed (`uv tool install`); `preflight._warn_cockpit_not_on_path` soft-warns when it isn't (never hard-fails — `statusline`/`starship` run fine via `-m cockpit.cli`). Add new entry points as `cockpit <sub>` subcommands in `cli.py`, not new file-path invocations.

**`cockpit watch` is a Textual TUI, and the TUI *is* the daemon (`cockpit/tui/`).** No headless watch fallback and no `once` mode — a non-TTY `watch` exits 2 (requires a terminal). The app owns the pidfile (`daemon.claim_pidfile` before `app.run()`, `release_pidfile` on unmount), runs the slow/fast ticks in `@work(thread=True)` workers behind per-tick phase gates ("idle" → "waiting" → "running"). The tick bodies (`cockpit.cockpit._once_with`/`_fast_tick`) are **lock-free**; the app serializes slow vs fast under its own `_tick_lock`, acquired *inside* the worker — so a tick blocked acquiring the lock shows "waiting" in the header, distinct from "running" (holds it). It routes signals via `loop.add_signal_handler` (SIGUSR1 → slow kick, for external `kill -USR1`; SIGTERM/SIGHUP → clean exit) — never `signal.signal`, which raises off the main thread. **Startup is slow-first:** on mount it kicks only the slow tick; the fast loop starts (`_start_fast`) after the first slow completes, so the first fast republish isn't a no-op against empty PR caches. The center is a navigable worktree table (`cockpit/tui/widgets/worktree_table.py` — a `DataTable` with a row cursor, keyed by worktree path); the repo is conveyed by tinting the workspace name with its `sidebar_color` (reusing `CMUX_COLOR_ANSI`), not a column. The table is **read-only** — it only reads the same flat cache cells starship reads (a muted PR prefixes its workspace name with 🔇, read from the daemon-written `pr-muted` cell; a coworker PR shows `@login` from the daemon-written `pr-author` cell, and the 💬 column shows the unaddressed-review-thread count from the daemon-written `pr-comments` cell — all three written by `cache.py::_write_pr_flat_cells` and republished by the fast tick), preserving the daemon-is-sole-writer invariant below. The row actions (`f` focus, `p` open-PR, `l` open-Linear, `c` close, `C` force-close, `m` mute/unmute, `N` nudge, `n` new) live on the *app*, not the table, and (except `n`, which spawns) never write a cache cell: `f` resolves the cursor row (`WorktreeTable.current_path` → cwd→path workspace ref) and calls `cmux focus` (cmux-only); `c` runs the `probe_blockers` gate (refuses on dirty/unpushed/open-PR) then `enqueue`s a `TeardownRequest` and kicks the slow tick, so teardown still flows through the daemon's `_drain_close_requests` (the daemon stays sole writer); `C` is the force variant — it overrides the *soft* open-PR blocker (`forced=True` on the request) but still refuses on the *hard* `worktree_state_blockers` (uncommitted / unpushed work), so force can never discard local work. Closing is TUI-only — there is no `cockpit close` CLI; the daemon's autoclose/orphan-reap paths still enqueue teardowns directly through `orchestrators.teardown`; `m` toggles the row PR's nudge mute by writing a `NudgePref` (all categories, no expiry — the same full mute `cockpit nudge mute` writes), which is session state and NOT a cache cell, then kicks the slow tick so the daemon republishes the `pr-muted` cell + pills and the 🔇 glyph catches up within the cycle; `N` sends a nudge to the row's workspace immediately via `cmux.nudge_if_idle` (no `pr_number`/`category`, so it overrides the mute + slow-tick throttle a deliberate keypress is meant to bypass) but still honours that gate's idle/parked safety check, so it never types into a running turn or a pending permission prompt — a passive skip-with-toast when the session isn't at rest, never a forced `send`. `n` opens a text-box modal (`NewWorkspaceScreen`) and, on submit, fires `cockpit new <source>` detached via module dispatch (`python -m cockpit.cli new`, NOT `spawn.py` by path — path invocation breaks the package imports) — the same path `/cockpit:new` and the daemon's `_bg_spawn_pr` walk — so the TUI never blocks on `git fetch` + worktree add; the typed value is auto-detected by spawn.py (bare name → new branch, `#N`/URL → PR, Linear id → ticket), and the repo selected in the modal — a `Select` shown when more than one repo is configured, defaulting to the cursor row's repo — is passed as the child `cwd` so a bare name routes to the chosen repo (with a single repo there's nothing to pick and that sole repo's path is used), and the slow tick is kicked so the new worktree surfaces as a row on a later cycle (inventory is derived, not stored — `spawn.py` itself never writes a cache cell, so the daemon stays sole writer). `N` moved off `n` precisely so the more-frequent New action gets the bare key. Each runs in a `@work(thread=True)` worker since they shell out to git/cmux/spawn (or, for `p`/`l`, open a browser). Five further *global* (not per-row) bindings round out `BINDINGS`: `s` sync (manual slow-tick kick), `r` repo-config screen, `o` output pane, `q` quit, and `u` self-update. `u` self-updates: the hourly `_check_update` worker compares `version.running_version()` against `version.latest_version()` and, when newer, paints an "⬆ update available" indicator in the header; pressing `u` then `self.exit(return_code=RESTART_EXIT_CODE)` (42) — `cockpit.cockpit._watch` returns `app.return_code`, and the `bin/cockpit.sh` supervisor loop catches that exact code, runs `bin/update.sh`, and relaunches `cockpit watch`. The update can't take effect in-process (it `uv tool install --force`s the very package the daemon runs from), so the restart is intrinsic, not cosmetic; `u` with no advertised update is a no-op toast. `bin/update.sh --check` exposes the same version comparison for scripts (exit 10 = update available, 0 = current). Every tick `print` is captured through one process-wide stdout/stderr `_QueueWriter` (never per-tick `redirect_stdout` — the two tick threads would race) so prints can't corrupt the screen; the log pane that displayed them is temporarily removed (drained-and-discarded), so re-adding a `LogPane` to `compose` restores it. `lib/daemon.py::run_watcher` (the old headless loop) is retained only as the tested signal/pidfile primitive; it no longer backs `watch`.

**Only the daemon writes to the cache.** Renderer field printers in `cockpit/lib/starship.py` are strictly read-only — no `gh`, no `git`, no subprocess forks, no atomic_write calls. The daemon owns every cell:

- **Slow tick** (`slow_poll_interval_seconds`, default 300s) — `cockpit/orchestrators/cycle.py::cycle_all` runs the full reconcile (gh PR fetch, base-distance fetch, per-PR JSON, branch-keyed PR flat cells, base-distance/ahead cells, git-state cells, pills).
- **Fast tick** (`fast_poll_interval_seconds`, default 30s) — `cockpit/cockpit.py::_fast_tick` does network-free republishing: git-state cells for every worktree (via `write_git_state_cache`) and PR flat cells from the persistent JSON snapshots (via `republish_pr_caches_from_disk`). The fast tick exists so `git checkout` and OS-level tmpdir wipes recover within ~30s instead of ~300s. Both ticks serialize on the TUI app's `_tick_lock` (`cockpit/tui/app.py`) so they never collide.

When introducing a new cell, the writer goes in `cockpit/lib/cache.py` and the call site goes in the daemon's slow tick (decision + snapshot) and/or the fast tick (republish from snapshot). Never let a renderer path consult source state directly — that produces same-render disagreement between fields (one segment reads the snapshot, another reads live state) and was the bug class this design eliminates.

**`sidebar_color` is a cosmetic, cmux-only per-repo field.** The daemon applies it in the slow tick via `_apply_repo_colors` (`cockpit/orchestrators/cycle.py`), which calls `cmux workspace-action --action set-color` for each workspace the repo owns. Deduplication runs through `pill_state` under a `color:<ref>` key — cmux is only touched when the color changes or after a daemon restart. Validation happens at preflight (`cockpit/lib/preflight.py::_validate_sidebar_colors`) and hard-fails with `sys.exit(2)` on an unknown color name, so invalid values never reach the slow tick. The same value also tints the repo's `owner/name` in the cycle log via `_repo_name_color`, which maps the name through `colors.CMUX_COLOR_ANSI` — that dict is the single source of truth for the valid set (`cmux.WORKSPACE_COLORS = frozenset(CMUX_COLOR_ANSI)`).

**Workspace names track the worktree's branch (`wt.label`), re-asserted on both ticks.** A workspace is named `wt.label` at spawn, but the name drifts — the user renames it, a closed-then-reopened PR reuses the branch, or a limux spawn lands a uuid. `wt.label` (`cockpit/lib/git.py::branch_label`) derives from the *branch*, not the dir basename (`wt.short`): it strips the repo's configured `branch_prefix` (e.g. `khivi/`), drops a leading base-branch segment (`master/`, `main/` — `khivi/master/fnox-age` → `fnox-age`), slugifies the remainder, then drops a leading ticket/PR token (`pe-4608-`, `123-`) so the label reads as the human description — but never to "" (a bare-ticket branch like `khivi/pe-4516` keeps its id). So a dir named `pe-4516` holding branch `khivi/pe-4608-understand-dag-builder` labels `understand-dag-builder`, not `pe-4516`. The prefix is threaded onto each `Worktree` at construction (`worktrees`/`worktrees_basic` take a `branch_prefix` arg, sourced from `repo_entry["branch_prefix"]`); `short` stays the dir basename for cwd→path matching and `git worktree remove`. cockpit resolves workspaces by cwd→path (never by name), so drift was silently tolerated until the user had to run `cmux rename-workspace` by hand. `cmux.rename_workspace_if_needed` re-asserts the convention (idempotent: a no-op when the name already matches), called per-tracked/orphan workspace in the slow tick (`cycle.py::_refresh_tracked_pills`, `_refresh_orphan`) and in bulk in the fast tick (`cmux.reconcile_workspace_names`, cwd→path matched) so a drift recovers within ~30s. The cwd→worktree *name* fallbacks (`find_cockpit_workspaces`, the cycle `wt_by_name` maps) and both daemon spawn names (`spawn_pr_workspace`, `spawn_orphan_workspace`) key on `wt.label` too, so resolution and relabel agree. Like `sidebar_color` it is a cosmetic cmux-only output, not a render cell, and never a `send`. **Consequence**: a manual rename that doesn't match the branch label is reverted to `wt.label` on the next tick — to get a custom sidebar label, rename the branch, not the workspace. A label collision (two branches whose post-strip slug matches) is cosmetic only: cmux resolution is cwd→path, so the correct session still receives its pills. **Exception — the primary checkout is exempt.** A workspace whose cwd is the repo's canonical clone (`wt.is_primary`, on a main branch) keeps its custom name: the slow-tick paths already skip it (no PR → never `tracked`; `master` ∈ `MAIN_BRANCHES`), and `reconcile_workspace_names` guards on `wt.is_primary`. Without this, a workspace parked on `master` to run skills (e.g. one the user named `morning` for morning-align) was force-renamed every fast tick, and the "rename the branch" escape hatch couldn't apply — the user won't rename a main branch. `rename-workspace` is not a pill verb, so it works on cmux and limux alike. The log line fires only when a rename actually happens (divergence), never on the steady-state no-op.

**The daemon creates worktrees in the background — it never blocks the tick on `git`.** Two cases in `cycle.py::_spawn_missing_workspaces` shell out via module dispatch (`python -m cockpit.cli new`, the same path `/cockpit:new` walks) in a detached `subprocess.Popen(start_new_session=True)`, so `git fetch` + worktree add never stalls the reconcile; the new worktree surfaces as cells on a later cycle (inventory stays derived, not stored).

- **My PR with no worktree** → background `cockpit new --pr <n> --repo <name>` (creates worktree + workspace with the plan-only prompt). Replaces the old "create one with /cockpit:new" warning; always on.
- **`review_prs: true|false` (per-repo, default false)** → for *every* other-authored open PR without a local worktree, background `cockpit new --pr <n> --review` (worktree + workspace whose first turn runs `/review`). Discovery uses `gh.list_open_pr_heads` (a paginated, uncapped `is:pr is:open` search — the daemon's normal query is `author:self` + per-worktree aliases, so coworker PRs are otherwise invisible). Uncapped by design: a busy repo can spawn many review worktrees at once; every spawn is logged. Validated as a bool at preflight (`_validate_review_prs`).

Both paths run through `_bg_spawn_pr`, which keys an in-flight guard in `pill_state` (`spawn:<owner>/<name>:<branch>`, a `time.monotonic()` stamp expiring after `_SPAWN_INFLIGHT_TTL_SECONDS`) so a SIGUSR1 kick can't double-launch while a creation is mid-flight, and a failed creation is retried on a later tick. Detached stdout/stderr land in `$COCKPIT_HOME/spawn.log`. Once a review worktree exists it is tracked via the normal `match_worktrees` coworker path and torn down on merge/close like any other; orphan auto-spawn stays `{self_user}/`-prefix gated, so review worktrees are never orphan-spawned.

**The single exception is session-scoped cells.** `lib.claude.stash_from_stdin` writes `context-<sid>`, `rate-limit-5h-<sid>`, `model-<sid>`, `permission-mode-<sid>`, `transcript-path-<sid>`, and `cost-<sid>` from Claude Code's statusLine stdin in the statusline path (not the daemon). These cannot route through the daemon because the data only exists in the real-time statusLine stream. They are session-scoped and never read by the daemon, so the rule "renderer reads, daemon writes" is preserved for everything the daemon owns. Do not extend this exception to any new cell.

**The nudge idle-gate trusts the persistent `idle=` pill, NOT cmux's native `claude_code=Needs input`.** `nudge_if_idle` (`cockpit/lib/cmux.py`) must distinguish "parked at the main prompt (safe to `send` a nudge)" from "awaiting a y/n permission decision (unsafe — the nudge text would land in the confirmation)". cmux's native `claude_code=` has three values, but only two are usable: `Running` (active) and `Idle` (Stop fired, parked). The third, `Needs input`, is **ambiguous** — it fires both for an idle-at-prompt session aged past Claude's ~60s Notification *and* for a pending permission request mid-turn — so it is never a safe at-rest signal on its own (verified against the cmux event stream). The gate therefore: blocks on native `Running` (also catches a dropped `idle=` clear left on a now-running session); treats the workspace as safe iff the `idle=` pill is present OR native is the unambiguous `Idle`; and self-heals a dropped Stop-hook write by re-asserting `idle=` when native `Idle` holds but the pill is missing. The `idle=` pill itself (`hooks/cmux-idle-pill.sh`, Stop branch) is set with a verify+retry loop — its silent loss under cmux-daemon contention (`Broken pipe`) was the original bug, leaving genuinely-parked workspaces un-nudgeable forever. Do not "simplify" the gate to trust `Needs input`; that reintroduces the permission-prompt hazard.

**The `devdone=` pill marks a PR whose Linear ticket(s) reached the dev-done column — Linear is the one auxiliary state source.** Gated on the repo being Linear-configured (`linear_keys`) AND the PR's body carrying a `Linear: [PE-1234](url)` footer. Delivery is resolved **footer-only** (`lib.linear.parse_linear_footers`) — never the branch-slug regex, which catches predecessor / follow-up tickets the PR doesn't deliver. This is the same strict signal the `morning-align` skill's `linear_delivery.py` uses; `extract_ticket(branch)` stays the footer's id pill but is NOT a delivery signal. The resolved `{tickets:[{id,state}], fetched_at}` block is **cached in the PR JSON** (`write_pr_cache`) like any other network round-trip: `cycle.py::_resolve_linear_block` refetches each ticket's workflow state from the Linear GraphQL API (`lib.linear.fetch_ticket_state`, `LINEAR_API_KEY` env) only when the footer id-set changes OR the block ages past `linear_state_ttl_seconds` (default `3 × slow_poll_interval_seconds`) — the TTL backstop catches a state move that doesn't touch the footer. `cycle.py::_track_dev_done` then reads that cached block (no network) and raises the green pill via `cmux.apply_devdone_pill` only when *every* delivered ticket is in `linear_dev_done_state` (default "Dev Done"). It is a passive slow-tick visual, never a `send`, and absent from `cmux.ACTIONABLE_KEYS`. Missing `LINEAR_API_KEY` degrades silently to no pill (preflight warns once). Do NOT add a per-cycle Linear issues query / branch-match signal without revisiting the cache-and-refresh design here.

**`linear_done_on_merge` is the one sanctioned exception to "Linear is read-only auxiliary" — opt-in, the daemon's only Linear *write*.** Default `false`; resolved per-repo over the top-level global (`config.linear_done_on_merge`), so you enable it only where you own the tickets. When on, the slow tick's `cycle.py::_transition_merged_tickets` moves a merged PR's delivered Linear tickets to `linear_merge_done_state` (default "Done", distinct from the `devdone=` pill's `linear_dev_done_state`) via the `lib.linear.update_ticket_state` → `issueUpdate` mutation — the single GraphQL write in the codebase. It fires on the same `_is_post_merge_stale` signal `_maybe_autoclose` uses but *independently of teardown*: a merged PR ships its tickets even when the worktree is held back (uncommitted work, unaddressed threads). Per ticket (read footer-only from the cached PR JSON's `linear` block — no extra discovery query): skip unless assigned to the API key's `viewer` (never move a teammate's ticket); skip if already at target or its state `type` is `canceled` (never resurrect — note "Dev Done"/"Done" are *both* `type: completed`, so name-equality, not type, decides "already done"); else resolve the target state's UUID for that ticket's team (`fetch_team_states`, cached per cycle) and mutate. A per-run `merged-done:<owner>/<name>:<ticket>` marker in `pill_state` stops a kept-but-merged worktree from re-querying every tick; the already-at-target name check is the cross-restart backstop, and a failed mutation clears the marker to retry. Gated on `linear_keys` + `LINEAR_API_KEY` + not-dry; missing key degrades silently (preflight warns once). It is a slow-tick action, never a `send`. Before extending the daemon's Linear surface to any *other* write, treat this as the precedent: opt-in, viewer-gated, idempotent, logged.

# One-time after cloning — wires pre-commit hooks for commit + push stages:
./setup.sh

## Release versioning

Handled by the pre-push hook — bumps `.claude-plugin/plugin.json`'s `version` automatically. No manual action needed.

## Test layout

New modules get their own `test_<name>.py` — don't append tests for a new source file to an unrelated test module. Shell hooks under `hooks/` are the exception: they live as `tests/test_<hook>.py` with no Python source mirror.

## Test style by layer

- **Leaf modules** (`cockpit/lib/*` wrapping `git`, `gh`, `cmux`, `shutil.which`, `subprocess.run`, etc.) test against the real tool on `tmp_path`. Stubbing the underlying command tests the stub, not the integration.
- **Orchestrators** (`cockpit/orchestrators/*`) compose those leaves. Tests mock collaborator calls (`patch.object(teardown_mod, "remove_worktree", …)`) to assert ordering, guards, and gating without re-validating the leaves underneath.
- **CLI entry-points** (`tests/test_<script>.py` for `cockpit/{cockpit,spawn}.py`, plus `tests/test_cli.py` for the dispatcher) test the argparse / routing layer. Mock at the orchestrator boundary (`patch("cockpit.cockpit.teardown", …)`) or, for the dispatcher, at each module's `main` — the layer below is covered by its own tests, so re-exercising it here adds noise.
- **TUI** (`tests/tui/*`) drive the Textual app headlessly via `App.run_test()`/Pilot (no TTY) with the tick functions and `load_config` injected; card markup is a pure function tested off seeded cache cells. Test the TUI's own scheduling / in-flight gating / log capture, not the reconcile cycle underneath.
- **End-to-end** (`tests/e2e/*`) run the full pipeline against real binaries. No mocking. These are the slowest and most fragile — reserve for genuinely cross-layer behavior (e.g. `cship` + `starship` integration).
